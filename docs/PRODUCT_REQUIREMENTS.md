# 产品需求（FROZEN）

## 1. 产品定位

`reboot-health` 是面向个人健康管理、训练计划、恢复管理、饮食计划、执行反馈和长期行为改善的 AI-first 产品。

产品通过自然语言和必要的结构化界面协助用户：

- 记录并纠正健康事实、目标、偏好和限制。
- 基于已确认事实生成候选综合健康计划。
- 解释建议、风险、来源和不确定性。
- 记录训练、恢复和饮食执行、历史修订和用户决定。
- 处理体检报告、图片和其他健康文件。

本产品不做医学诊断，不替代医生、急救服务、营养师或其他持证专业人士。

## 2. 用户和客户端

第一阶段允许单个真实用户正式使用，但所有业务数据必须具备 `userId` 归属边界，为未来多用户、家庭和团队模式保留扩展能力。

正式客户端与实施顺序：

| 客户端 | 定位 | 顺序 |
|---|---|---|
| 独立 Web 用户端（规划为 `clients/web/`） | Phase 4 首个普通用户业务验证入口 | 首先实现 |
| 微信小程序 | 面向最终用户的主要移动入口 | Web 闭环稳定后 |
| Flutter Android/iOS | 与 Web/小程序保持同一业务能力和协议 | Web 闭环稳定后 |
| Vue 3 Web（`frontend/`） | 管理、运维、任务诊断和审计；不代替用户确认业务计划 | 独立演进 |

所有客户端只调用 Health Platform。任何客户端均不得直接调用 `health-agent`、数据库、Redis、MinIO 或内部 Tool handler。

`frontend/` 不得同时承担普通用户健康业务和管理端职责。

## 3. 核心产品原则

1. **事实优先**：缺少关键事实时进入 `WAITING_USER_INPUT`，明确说明缺什么、为什么需要，禁止静默猜测。
2. **模型输出是候选**：模型不得把推断、摘要、OCR 结果、饮食估算或建议直接写成正式健康事实。
3. **用户拥有最终决定权**：候选计划、风险确认、历史事实纠正和重要调整必须由用户明确决定。
4. **动态 Agent，不是固定流水线**：主 Agent 根据任务决定直接回答、查询 Tool、澄清、委派 Sub-Agent 或生成候选结果。
5. **确定性边界**：身份、权限、Readiness、版本、幂等、状态转换、风险底线、Sandbox、Secret、审计和数据一致性由程序保证。
6. **可恢复执行**：用户关闭页面、网络断开或 Worker 重启后，任务仍可继续或安全恢复。
7. **可追溯**：Program、Plan、Fact、Risk、File、Task、Tool 和营养数据必须保留来源、版本、时间和审计关系。
8. **低负担执行**：默认采用轻量记录，只有在安全、分析或调整确实需要时才追加细节。

## 4. 阶段边界

### 4.1 Phase 3

Phase 3 负责生产基础、Identity、用户归属授权、服务间授权、health-agent 生产执行、Secret、日志脱敏、跨用户和故障恢复验收。

Phase 3 不负责训练、恢复、饮食及其执行反馈业务。

### 4.2 Phase 4

Phase 4 负责第一个完整健康业务闭环。详细基线见 [`PHASE_4_BASELINE.md`](PHASE_4_BASELINE.md) 和 ADR 0023。

Phase 4 当前状态为 `DISCOVERY`，未批准的 Slice 不得开始代码实现。

## 5. 核心业务闭环

```text
用户自然表达目标或问题
→ Health Platform 创建持久 Task
→ health-agent 动态识别信息和 Tool 需求
→ 缺少关键事实时针对性追问
→ 生成 FactCandidate / GoalCandidate
→ 用户分级确认
→ 生成 HealthProgramCandidate
→ 用户分领域查看训练、恢复和饮食
→ 用户整体确认 ProgramVersion
→ Health Platform 使综合计划正式生效
→ 用户记录执行和变化
→ 事件反馈、每日 Check-in 和周期复盘
→ 系统评估偏差和风险
→ 生成 AdjustmentCandidate
→ 用户确认新的 ProgramVersion
```

底层执行是异步、持久、可恢复的；前端可以通过 SSE 实时显示进度。

## 6. 对话和任务体验

- 用户提交消息后页面可以持续等待并显示流式进度。
- 用户离开页面或断网不取消 Task。
- 重新进入时自动恢复同一 Task、进度和上下文。
- 用户可以在 Task 运行中继续发送补充信息。
- 当前不可中断的模型或 Tool 步骤先安全结束；之后按消息顺序批量处理未消费信息。
- 用户操作只有“终止当前执行”，不提供“暂停”按钮。
- `WAITING_USER_INPUT` 是正常 Run 状态，不等于用户暂停。
- 用户终止后，未确认候选进入 `ABANDONED`，可复用的只读结果和审计仍保留。

## 7. 对话式信息采集

首次信息采集采用纯对话，不以传统必填问卷作为主流程。

```text
用户自然描述
→ Agent 提取候选事实和目标
→ ReadinessPolicy 判断缺失、冲突和风险
→ Agent 只追问当前必要信息
→ 用户确认
```

系统内部必须维护：

- `TRAINING_READY`
- `RECOVERY_READY`
- `NUTRITION_READY`
- `PROGRAM_READY`

缺少阻断型关键信息时，不得生成可确认的综合计划。

用户界面可以使用卡片、摘要、选择器或其他结构化组件帮助确认和查看，但不能把固定表单伪装成 AI-first 对话。

## 8. Fact、Goal 和历史纠正

- Health Platform 是业务事实唯一权威。
- Agent 只创建 `FactCandidate` 和 `GoalCandidate`。
- 普通、低影响事实可以在一个对话阶段结束时批量确认。
- 疾病、疼痛、损伤、药物、食物过敏、关键身体指标和目标数值等高影响事实必须单独确认。
- 文件提取的健康字段仍必须逐项确认。
- 模型推测不得伪装成用户明确陈述。
- 新事实与旧事实冲突时，后确认且有效的事实成为当前值，旧值标记为 `SUPERSEDED`。
- 用户可以纠正历史事实，但必须二次确认。
- 纠正必须展示原值、新值、来源和可能影响。
- 旧值保留在修订历史中，不做无痕覆盖。
- 事实修正影响当前计划时，用户选择保留、重新分析或重新生成；系统不得静默覆盖当前版本。

## 9. HealthProgram 产品语义

### 9.1 综合计划结构

```text
HealthProgram
└── ProgramVersion
    ├── TrainingPlanVersion
    ├── RecoveryPlanVersion
    └── NutritionPlanVersion
```

三个子计划独立版本化，但最终由一个 `ProgramVersion` 保持组合一致性。

### 9.2 时间结构

```text
ProgramCycle
→ ProgramPhase
→ WeekPlan
→ DailySchedule
```

- 生成完整周期框架。
- 只详细生成近期计划，并随执行滚动展开。
- 首版目标方向为未来两周 WeekPlan 和未来七天 DailySchedule；具体窗口由后续详细设计配置。

### 9.3 候选与确认

- Agent 生成的是候选综合计划。
- 用户按领域查看训练、恢复和饮食。
- 用户可以对某个领域提出修改。
- 最终必须整体确认当前 `ProgramVersion`。
- “确认”即发布并生效，不再增加第二个发布按钮。
- 新版本未确认前，旧的当前版本继续有效。
- 确认页必须展示主要事实来源、时间范围、风险、估算、不确定性和模型切换信息。

### 9.4 修改、修订和执行

- 用户界面只突出当前综合计划，后台保留内部版本和变更事件。
- 已完成的执行记录是历史事实，不因计划修订而改变。
- 修改只影响尚未完成的未来部分。
- 用户明确提出的低风险、等价修改通过确定性检查后可以立即应用并保留 Revision。
- 训练负荷、热量目标、阶段结构和安全限制等实质性修改必须重新生成候选。
- Agent 主动提出的修改仍是候选，需要用户整体确认。
- 跨领域变化必须重新执行一致性检查。

## 10. 执行记录和反馈

### 10.1 训练

默认记录：

- 完成、部分完成、跳过或中止。
- 实际强度。
- 疼痛、不适或其他异常。

用户可以展开记录每组重量、次数、RPE 和动作替换。

### 10.2 恢复

默认记录：

- 睡眠时长与质量。
- 疲劳、酸痛、压力和精神状态。
- 恢复任务完成情况。

### 10.3 饮食

默认记录：

- 餐次完成度和主要偏差。
- 饥饿感、饱腹感和异常反应。

用户可以展开记录食物、份量和营养数据。

### 10.4 反馈节奏

- 任务完成后的事件反馈。
- 低负担每日 Check-in。
- 周期复盘，首版以周复盘为默认方向。
- 疼痛、异常疲劳、连续未完成或趋势异常时即时追问。

## 11. 调整

调整触发来源包括：

- 用户主动请求。
- 周期复盘。
- 连续执行偏差。
- 疲劳、疼痛或睡眠异常。
- 体重、围度或其他目标趋势偏离。
- 阶段结束。
- 关键健康事实变化。
- 滚动计划窗口即将结束。

系统可以自动评估影响范围，但只能生成 `AdjustmentCandidate`。未经用户确认，不得让新的长期 `ProgramVersion` 生效。

紧急风险可以先暂停受影响的未来任务；暂停不等于自动发布新计划。

## 12. 食物与营养数据

营养数据来源必须标记：

- `VERIFIED_DATABASE`
- `LABEL_DECLARED`
- `USER_DECLARED`
- `AI_ESTIMATED`

AI 估算必须展示：

- 参考值。
- 合理范围或上下界。
- 置信度。
- 关键假设和主要不确定因素。

低置信度的单次估算不得独立触发明显的营养目标调整。

标准食物数据库、许可证、地区覆盖、菜品映射和更新机制必须先完成 Spike，不得由实现 Agent 随意选择。

## 13. 风险和人工决定

风险采用确定性规则分级，Agent 负责上下文识别、解释和追问：

- `R0`：无明确风险。
- `R1`：提醒。
- `R2`：限制。
- `R3`：暂停并补充信息。
- `R4`：阻断并建议寻求专业帮助。

风险作用域为 `TRAINING`、`RECOVERY`、`NUTRITION` 或 `GLOBAL`。

- Agent 可以提出更高风险候选，但不得降低确定性规则结果。
- 医学阈值未完成专业审核时标记 `NEEDS_MEDICAL_REVIEW`，不得自行实现。
- 已完成行为只生成风险发现和提醒，不篡改原执行记录。
- 所有风险提示、用户确认、最终选择和影响项必须审计。
- 技术安全、身份越权、跨用户访问、Secret 泄露、数据损坏和无法确认的副作用必须 fail-closed。
- 可能涉及紧急医疗状况时，产品必须突出建议联系当地急救或专业人士，不输出诊断结论。

## 14. 提醒和通知

计划提醒采用确定性调度：

```text
Confirmed DailySchedule
→ ReminderTask
→ 调度器按照时间、时区、免打扰和重试规则触发
→ Agent 生成不改变任务语义的个性化文案
```

必须支持：

- 用户修改提醒时间。
- 时区和免打扰。
- 延后、跳过、过期和有限重试。
- 幂等与防重复。
- 连续忽略后的降频。

Agent 不得绕过通知偏好、提高频率或通过文案修改计划和安全边界。

系统通知还包括：

- Task 完成、失败或需要用户处理。
- Program 摘要和待确认调整。
- 风险摘要。
- 管理员高优先级告警。

通知不得承载完整敏感健康内容；用户必须回到正式客户端完成确认。第一版不接受邮件回复、邮件指令或邮件确认。

具体通知渠道和供应商不在当前基线中冻结。

## 15. 文件和报告

正式支持上传：

- 图片。
- PDF 体检报告。
- 其他明确允许的健康相关文件。

要求：

- Health Platform 管理身份、配额、元数据、业务关联和删除。
- 原始文件存入 `ObjectStorageProvider`；第一版实现为 MinIO。
- Agent 仅通过受控 Tool 获取短期只读访问。
- PDF、图片和不可信文件解析必须进入 Sandbox。
- Agent 提取的所有健康字段都必须由用户逐项确认后才能写入正式档案。
- 用户可申请彻底删除文件；删除范围包括原件、缩略图、OCR、解析产物、RAG Chunk、向量和临时副本，仅保留不含内容的删除审计。

文件能力的具体实施阶段由后续 Phase 计划决定，不因 Phase 4 基线自动进入实现。

## 16. Memory、上下文、RAG 和 Sub-Agent

- 完整用户可见对话保存在 Health Platform。
- `health-agent` 只保存执行、恢复和短期上下文所需数据。
- Conversation Summary 不是业务事实。
- 长期健康事实只能通过 Platform Tool 查询。
- RAG 只用于非权威历史上下文，不用于替代当前体重、当前 Program、用户档案或执行事实。
- 默认检索当前 Session；模型明确需要历史时才扩展到同一用户其他 Session。
- 禁止索引系统 Prompt、隐藏推理、Secret、未验证猜测、完整敏感 Tool 原始输出和其他用户数据。
- 主 Agent 可以按需创建 Sub-Agent。
- 第一版只允许顺序执行，一层委派；Sub-Agent 不得继续创建 Sub-Agent。
- Sub-Agent 只获得最小上下文、最小 Tool 范围和短期委派凭证。
- Sub-Agent 结果是结构化候选，不能直接发布 Program、写正式事实或代替用户确认。
- 主 Agent 负责接受、部分采用、返工或拒绝结果。

## 17. 管理端

Vue 3 管理端允许：

- 查看 Task、Run、Step、Tool、模型调用、失败、重试、资源和审计摘要。
- 查看告警、风险处理状态和同步异常。
- 重试、恢复或终止异常任务。
- 暂停异常执行。
- 撤销 Session 或禁用账户。
- 标记需要人工复核的风险事件。
- 发起待用户确认的数据修正请求。

管理员不得：

- 修改用户健康事实和 Program 内容。
- 代表用户确认 Program 或风险。
- 修改历史执行记录。
- 降低确定性风险等级。
- 编辑 Conversation Summary。
- 把失败任务标记为虚假成功。
- 无必要地读取完整健康原文。

所有管理操作必须记录操作者、原因、时间、原状态、动作和结果。

## 18. 生产验收

必须同时通过以下五类验收，才允许标记为可生产使用：

1. **业务闭环**：自然对话 → 事实和目标确认 → 综合计划候选 → 整体确认 → 执行 → 反馈 → 调整候选 → 新版本确认。
2. **故障恢复**：API、Worker、Redis、PostgreSQL Pod 故障时任务不丢、重复受控、可恢复；调整失败时旧计划继续有效。
3. **安全验收**：mTLS、短期 JWT、Tool Scope、Secret、Sandbox、跨用户和跨任务阻断全部通过；管理员不能代替用户决策。
4. **数据一致性**：Outbox/Inbox、重复与乱序事件、版本、执行历史、营养来源、修订和纠错有自动化验证。
5. **运维验收**：灰度、滚动升级、迁移、告警、备份恢复、提醒幂等和故障演练有可审计记录。

## 19. 非目标

- 不做医学诊断、处方或替代专业医疗服务。
- 不允许模型绕过 Health Platform 直接写数据库。
- 不允许客户端直接访问 health-agent。
- 不开放任意 Shell、任意文件系统、任意 SQL 或无限制网络访问。
- 不把 Kubernetes、数据库副本或 Hyper-V VM 宣称为跨物理故障域高可用。
- 第一版不建设社区、排行榜、商城、支付、公开第三方 Agent 市场或邮件入站自动化。
- 当前基线不决定具体 Web 技术栈、食物数据库、医学阈值、训练动作库和通知供应商。

## 20. Identity 第一版（Phase 3 当前范围）

- 多用户、单租户；邮箱与用户名全局唯一并支持不区分大小写登录，展示名称独立。
- 未验证邮箱可登录受限账号；核心健康能力要求邮箱已验证。
- 密码使用 Argon2id、最少 12 位，拒绝常见与已泄露密码；管理员和客服不能设置用户密码。
- Web/Flutter/Vue 使用邮箱或用户名 + 密码，以及 Authorization Code + PKCE；小程序使用微信 code 换取 Platform 自有会话。
- 多设备独立 Session/Refresh Token Family；用户可撤销单设备、其他设备或全部设备。
- 普通用户 MFA 可选，高权限角色强制 TOTP + 一次性恢复码。
- 支持邮件验证、密码恢复、固定安全问题辅助、数据导出任务与 7 天冷静期账号注销。
- 第一版人类账号角色仅包含 `USER` 和 `ADMIN_OPERATOR`；后者只执行确定性身份管理与故障处置，且不自动获得健康原文或健康业务修改权限。`SERVICE_HEALTH_AGENT` 是服务主体，不是人类账号角色。
