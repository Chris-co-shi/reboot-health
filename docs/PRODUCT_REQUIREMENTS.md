# 产品需求（FROZEN）

## 1. 产品定位

`reboot-health` 是面向个人健康管理、训练计划、执行反馈和长期行为改善的 AI-first 产品。

产品通过自然语言和结构化界面协助用户：

- 记录并纠正健康事实、目标、偏好和限制。
- 基于已确认事实生成候选 Plan。
- 解释建议、风险、来源和不确定性。
- 记录计划执行、历史修订和用户决定。
- 处理体检报告、图片和其他健康文件。

本产品不做医学诊断，不替代医生、急救服务或其他持证专业人士。

## 2. 用户和客户端

第一阶段允许单个真实用户正式使用，但所有业务数据必须具备 `userId` 归属边界，为未来多用户、家庭和团队模式保留扩展能力。

正式客户端：

| 客户端 | 定位 |
|---|---|
| 微信小程序 | 面向最终用户的主要业务入口 |
| Flutter Android/iOS | 与小程序保持同一业务能力和协议 |
| Vue 3 Web | 管理、运维、任务诊断和审计；不代替用户确认业务 Plan |

小程序和 Flutter 只调用 Health Platform。任何客户端均不得直接调用 `health-agent`、数据库、Redis、MinIO 或内部 Tool handler。

## 3. 核心产品原则

1. **事实优先**：缺少关键事实时进入 `WAITING_USER_INPUT`，明确说明缺什么、为什么需要，禁止静默猜测。
2. **模型输出是候选**：模型不得把推断、摘要、OCR 结果或建议直接写成正式健康事实。
3. **用户拥有最终决定权**：候选 Plan、风险确认、历史事实纠正和重要调整必须由用户明确决定。
4. **动态 Agent，不是固定流水线**：主 Agent 根据任务决定直接回答、查询 Tool、澄清、委派 Sub-Agent 或生成候选结果。
5. **确定性边界**：身份、权限、版本、幂等、状态转换、Sandbox、Secret、审计和数据一致性由程序保证。
6. **可恢复执行**：用户关闭页面、网络断开或 Worker 重启后，任务仍可继续或安全恢复。
7. **可追溯**：Plan、Fact、Risk、File、Task 和 Tool 结果必须保留来源、版本、时间和审计关系。

## 4. 核心业务闭环

```text
用户表达目标或问题
→ Health Platform 创建持久 Task
→ health-agent 根据上下文和 Tool 动态执行
→ 缺少关键事实时向用户追问
→ 读取已确认事实、历史和文件
→ 生成候选 Plan / 解释 / 风险发现
→ 用户逐项纠正事实并确认候选 Plan
→ Health Platform 发布正式 Plan
→ 用户记录执行和变化
→ 后续任务读取新事实并调整未完成部分
```

底层执行是异步、持久、可恢复的；前端可以通过 SSE 实时显示进度。

## 5. 对话和任务体验

- 用户提交消息后页面可以持续等待并显示流式进度。
- 用户离开页面或断网不取消 Task。
- 重新进入时自动恢复同一 Task、进度和上下文。
- 用户可以在 Task 运行中继续发送补充信息。
- 当前不可中断的模型或 Tool 步骤先安全结束；之后按消息顺序批量处理未消费信息。
- 用户操作只有“终止当前执行”，不提供“暂停”按钮。
- `WAITING_USER_INPUT` 是正常 Run 状态，不等于用户暂停。
- 用户终止后，未确认候选 Plan 进入 `ABANDONED`，可复用的只读结果和审计仍保留。

## 6. Fact 和历史纠正

- Health Platform 是业务事实唯一权威。
- 新事实与旧事实冲突时，后确认且有效的事实成为当前值，旧值标记为 `SUPERSEDED`。
- 用户可以纠正历史事实，但必须二次确认。
- 纠正必须展示原值、新值、来源和可能影响。
- 旧值保留在修订历史中，不做无痕覆盖。
- 事实修正影响现有 Plan 时只提示影响；用户选择保留、重新分析或重新生成，不自动暂停已确认 Plan。

## 7. Plan 产品语义

### 7.1 候选与确认

- Agent 生成的是候选 Plan。
- 用户可以整体确认，也可以局部修改后重新整体确认。
- “确认”即发布并生效，不再增加第二个发布按钮。
- 新 Plan 未确认前，旧的当前 Plan 继续有效。
- 候选确认页必须展示主要事实来源、时间范围、风险和模型切换信息。

### 7.2 修订和执行

- 用户界面只突出“当前 Plan”，后台保留内部版本和变更事件。
- 已完成的执行记录是历史事实，不因 Plan 修订而改变。
- 用户可以修改尚未完成的部分；完成部分保持原执行事实。
- 用户明确提出的低风险修改通过确定性检查后可以立即应用到未完成部分。
- Agent 主动提出的修改仍是候选，需要用户确认。

## 8. 风险和人工决定

- Agent 和系统识别、解释风险，提供证据和更安全替代方案。
- 模型识别出的潜在健康风险要求二次确认，但用户仍可在充分知情后决定未来 Plan 变化。
- 已完成行为只生成风险发现和提醒，不篡改原执行记录。
- 所有风险提示、用户确认、最终选择和影响项必须审计。
- 技术安全、身份越权、跨用户访问、Secret 泄露、数据损坏和无法确认的副作用必须 fail-closed。
- 可能涉及紧急医疗状况时，产品必须突出建议联系当地急救或专业人士，不输出诊断结论。

## 9. 文件和报告

正式支持上传：

- 图片。
- PDF 体检报告。
- 其他明确允许的健康相关文件。

要求：

- Health Platform 管理身份、配额、元数据、业务关联和删除。
- 原始文件存入 `ObjectStorageProvider`；第一版实现为 MinIO。
- Agent 仅通过受控 Tool 获取短期只读访问。
- PDF、图片和不可信文件解析必须进入 Sandbox。
- Agent 提取的所有健康字段都必须由用户**逐项确认**后才能写入正式档案。
- 用户可申请彻底删除文件；删除范围包括原件、缩略图、OCR、解析产物、RAG Chunk、向量和临时副本，仅保留不含内容的删除审计。

## 10. Memory、上下文和 RAG

- 完整用户可见对话保存在 Health Platform。
- `health-agent` 只保存执行、恢复和短期上下文所需数据。
- Conversation Summary 不是业务事实。
- 长期健康事实只能通过 Platform Tool 查询。
- RAG 只用于非权威历史上下文，不用于替代当前体重、当前 Plan、用户档案或执行事实。
- 默认检索当前 Session；模型明确需要历史时才扩展到同一用户其他 Session。
- 禁止索引系统 Prompt、隐藏推理、Secret、未验证猜测、完整敏感 Tool 原始输出和其他用户数据。

## 11. Sub-Agent

- 主 Agent 可以按需创建 Sub-Agent。
- 第一版只允许顺序执行，一层委派；Sub-Agent 不得继续创建 Sub-Agent。
- Sub-Agent 只获得最小上下文、最小 Tool 范围和短期委派凭证。
- Sub-Agent 结果是结构化候选，不能直接发布 Plan、写正式事实或代替用户确认。
- 主 Agent负责接受、部分采用、返工或拒绝结果。

## 12. 通知

第一版仅支持出站邮件：

- Task 完成、失败或需要用户处理。
- Plan 摘要。
- 风险摘要。
- 管理员高优先级告警。

邮件不得承载完整敏感健康内容；用户必须回到小程序或 App 完成确认。第一版不接受邮件回复、邮件指令或邮件确认。

## 13. 管理端

Vue 3 管理端允许：

- 查看 Task、Run、Step、Tool、模型调用、失败、重试、资源和审计摘要。
- 查看告警、风险处理状态和同步异常。
- 重试、恢复或终止异常任务。

管理员不得：

- 修改用户健康事实和 Plan 内容。
- 代表用户确认 Plan 或风险。
- 编辑 Conversation Summary。
- 把失败任务标记为虚假成功。

所有管理操作必须记录操作者、原因、时间、原状态、动作和结果。

## 14. 第一阶段生产验收

必须同时通过以下五类验收，才允许标记为可生产使用：

1. **业务闭环**：客户端提交 → Tool 查询 → 补充事实 → 候选 Plan → 用户确认 → 正式生效。
2. **故障恢复**：API、Worker、Redis、PostgreSQL Pod 故障时任务不丢、重复受控、可恢复。
3. **安全验收**：mTLS、短期 JWT、Tool 边界、Secret、Sandbox、跨用户阻断全部通过。
4. **数据一致性**：Outbox/Inbox、重复与乱序事件、RAG 失效、修订和纠错有自动化验证。
5. **运维验收**：灰度、滚动升级、迁移、告警、备份恢复和故障演练有可审计记录。

## 15. 非目标

- 不做医学诊断、处方或替代专业医疗服务。
- 不允许模型绕过 Health Platform 直接写数据库。
- 不允许客户端直接访问 health-agent。
- 不开放任意 Shell、任意文件系统、任意 SQL 或无限制网络访问。
- 不把 Kubernetes、数据库副本或 Hyper-V VM 宣称为跨物理故障域高可用。
- 第一版不建设社区、排行榜、商城、支付、公开第三方 Agent 市场或邮件入站自动化。
