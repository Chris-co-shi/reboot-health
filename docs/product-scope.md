# 产品范围

## 产品定位

`reboot-health` 是面向单个用户的 AI-first 健康、减脂和规律训练辅助系统。

产品核心不是让用户维护配置后台，而是让 AI 主动理解目标、限制和执行反馈，生成可解释的长期方向、阶段计划和每日行动；系统负责事实、安全、确认、审计和版本留痕。

本项目不做医学诊断，不替代医生意见。

## 工程定位

产品由三个核心部分协作：

- Python Health Agent Harness：智能与任务编排核心。
- Java Health Domain Kernel：事实、安全和领域状态权威。
- Flutter Client：正式用户体验。

项目的工程价值不只是调用模型，而是建设 Skills、Tools、Context、Memory、Approval、Trace、Evaluation 和 Recovery，使 Agent 在健康领域中可控制、可观察、可验证。

## 核心原则

- 用户主要通过自然语言表达现状、目标、偏好和不适。
- 用户负责纠正、确认和执行反馈，不维护内部枚举、版本号和技术字段。
- AI 从首次使用开始参与理解和规划。
- AI 输出默认是候选或草案，重要事实和计划必须进入确认流程。
- 确定性安全规则优先于模型建议。
- 普通日尽量自动获取数据，不能依赖每天定时录入。
- 训练日允许记录更详细的执行信息。

## 第一条产品闭环

第一阶段聚焦：减脂和体重管理、恢复规律训练。

```text
自然语言描述
→ AI 理解候选
→ 用户纠正与确认
→ 长期方向草案
→ 当前阶段草案
→ 首周计划草案
→ 安全检查与用户确认
→ 今日行动
→ 最小执行反馈
→ 用户记忆候选
```

M2.5-A 只建立客户端、AgentRun 和设备认证技术骨架；业务闭环在 M2.5-B、M2.5-C 分阶段实现。

## 交互方式

采用“自然语言 + 页面卡片”：

- 自然语言用于描述目标、补充状态、回答追问和理解原因。
- 页面卡片用于今日行动、AI 理解、计划草案、确认、反馈和趋势。

主导航方向：今日、教练、计划、数据、我的。

正式产品不向用户暴露 UUID、revision、PlanVersion、内部状态枚举或类型选择器。

## 自动化与确认

允许自动执行的低风险调整：调整当天顺序、降低训练量或复杂度、替换为恢复活动、缩短当天训练。自动调整只能降低风险，必须说明原因、可撤销并可审计。

必须用户确认：长期方向、当前阶段、新周计划、增加训练负荷、重要目标变化、重要健康约束变化和高风险建议。

## 平台范围

正式客户端使用 Flutter，目标平台为 iOS、Android、macOS 和 Windows。

第一阶段不开发独立 Watch App。后续 Apple 数据通过 iPhone 接入，Android 数据通过 Health Connect 接入。桌面端主要用于对话、计划审核、趋势和设备管理。

Vue 仅保留为冻结的内部调试工具。

## 数据输入原则

优先使用设备自动数据、执行过程数据和已有历史事实。只有在安全判断或连续偏离计划时，系统才提出一次一个关键问题。

允许训练提醒和周复盘提醒，不进行无上限日常催促。

## 当前不做

- 医学诊断。
- 未经确认自动发布长期计划或提高训练风险。
- 完整公开账号体系和运营后台。
- 社区、排行榜、商城和支付。
- 复杂多 Agent 自治。
- 在线训练基础模型。
- 在 M2.5-A 接入真实模型、健康设备同步、图片识别或训练播放器。
