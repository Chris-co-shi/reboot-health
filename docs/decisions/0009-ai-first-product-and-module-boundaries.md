# 0009 AI-first 产品与模块职责

## 状态

已确认。

## 背景

M2A 和 M2B 建立了档案、健康约束、目标和计划版本能力，但体验偏向配置工具。项目进入 Java、Python、Flutter 和 Vue 多运行时阶段，需要明确产品主线和模块职责。

## 决策

### 产品方向

- AI 从首次使用开始参与理解、澄清、规划和复盘。
- 用户主要负责自然表达、纠正、确认和执行反馈。
- 内部枚举、版本号和 revision 不作为主产品操作入口。
- 第一条业务闭环聚焦减脂、体重管理和恢复规律训练。
- 普通日尽量使用自动数据和历史事实，训练日允许详细记录。

### 交互和平台

- 正式客户端使用 Flutter，目标平台为 iOS、Android、macOS 和 Windows。
- 交互采用自然语言与页面卡片结合。
- Vue 冻结为内部调试工具。
- 第一阶段不开发独立 Watch App。

### 运行时职责

- Java：业务事实、状态机、确认、安全、审计、设备认证和 AgentRun 权威。
- Python：模型 Provider、编排和结构化候选输出，不持有业务事实。
- Flutter：正式用户交互和平台适配，只调用 Java API。
- Vue：内部数据检查和阻塞性修复。

### 交付方式

- M2.5-A：技术骨架。
- M2.5-B：首次 AI 规划闭环。
- M2.5-C：最小今日行动和执行反馈。
- 每个实施任务只能有一个主模块。
- 跨运行时改动先定义合同，再分别实施，最后单独集成验证。

## 影响

- Goal 继续作为目标唯一事实来源。
- HealthConstraint 通过候选与用户确认演进。
- 周计划继续由现有 PlanVersion 引擎发布。
- Python 不访问业务数据库，不直接发布计划。
- Flutter 不实现后端领域状态机。
- 新业务不在 Vue 和 Flutter 中双重实现。
- 仓库使用分层 `AGENTS.md` 限制任务范围。

## 非目标

- 不建设复杂多 Agent 自治。
- 不在当前阶段引入消息队列、向量数据库或工作流平台。
- 不建设公开多用户产品和运营后台。
