# 0014 Fact、Plan、风险与人工决定权

## 状态

已确认，2026-07-12 生效。

扩展 0003 和 0007。

## 背景

健康产品必须区分模型建议、用户确认事实、当前 Plan、历史执行和风险提醒。若模型可以直接把推断写成事实、静默修改 Plan 或覆盖历史记录，将失去可信性和可追溯性。

## 决策

- Health Platform 是 Fact、Goal、Plan、ExecutionRecord、Risk 和用户确认的唯一权威。
- 模型、RAG、Summary、OCR 和文件解析结果默认是候选。
- 文件提取的健康字段必须逐项确认。
- 候选 Plan 经用户整体确认后即发布；新候选未确认前旧 Plan 继续有效。
- 已完成执行记录不可被 Plan 修订改变。
- 历史 Fact 可以纠正，但需二次确认，旧值保留并标记 superseded。
- Agent 识别的健康风险需要解释、证据、替代方案和高风险二次确认；用户决定未来非强制业务选择。
- 技术安全、越权、Secret、跨用户、非法状态转换和未知副作用必须硬阻断。

## 影响

正面：

- 模型不会成为不可审计的事实写入者。
- 当前 Plan、历史执行和纠正关系清晰。
- 用户可以理解并控制风险选择。

成本：

- 需要候选、revision、确认、风险 acknowledgement 和审计模型。
- 客户端必须处理 VERSION_CONFLICT 和候选失效。

## 约束

- 管理员不能代表用户确认 Plan 或风险。
- Agent Tool 只能创建候选、风险发现和变更请求。
- 具体医学阈值没有专业依据时必须标记 NEEDS_MEDICAL_REVIEW。
- 可能紧急的情况必须提示联系当地急救或专业人士，但不得输出诊断结论。
