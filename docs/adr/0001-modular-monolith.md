# ADR-0001: MVP 采用模块化单体架构

## 状态

已接受

## 背景

项目是个人使用的健康、减脂和体能重建应用。当前目标是尽快打通计划、执行、分析、AI 建议、用户确认和计划版本生成的闭环。

需求包含多个领域模块，但使用者只有一个人，部署目标是 Windows 10 + Docker + 本机 PostgreSQL 17。此阶段不需要独立扩展、独立部署或团队自治。

## 决策

MVP 采用模块化单体：

- 一个 Spring Boot 后端应用。
- 一个 Vue 前端应用。
- 一个 PostgreSQL 数据库。
- 后端按领域模块分包。
- 模块之间通过应用服务和领域接口协作。
- 不拆微服务。

后端模块预留：

- `profile`
- `goal`
- `plan`
- `execution`
- `metrics`
- `analysis`
- `rules`
- `ai`
- `adjustment`
- `audit`

## 备选方案

### 微服务

优点：

- 模块部署独立。
- 后续可按模块扩展。

缺点：

- 对个人项目过度复杂。
- 增加网络、部署、观测和数据一致性成本。
- 会拖慢 MVP 闭环落地。

### 单包 CRUD 应用

优点：

- 初期开发最快。
- 文件少。

缺点：

- 计划版本、规则引擎和 AI 调整边界容易混在一起。
- 后续很难保护关键不变量。

### 模块化单体

优点：

- 保持部署简单。
- 能表达领域边界。
- 支持后续按模块演进。

缺点：

- 需要持续维护模块依赖方向。
- 不能用服务边界强制隔离。

## 影响

更容易：

- 快速交付端到端闭环。
- 在单机环境部署。
- 用事务保护计划版本和调整确认状态。
- 通过代码结构表达领域边界。

更困难：

- 模块间隔离需要工程纪律。
- 未来如果拆服务，需要先稳定模块接口。

## 架构约束

- 领域规则不依赖控制器、数据库 Mapper 或 AI 客户端。
- AI 适配器只返回结构化建议，不直接创建计划版本。
- 计划版本创建必须由领域服务完成。
- 安全规则先于 AI 建议应用。
